# Debat Log — 2026-06-27l

**Debator:** Claude
**Scope:** Pre-implementation review for: (1) unique item/feature names, (2) Firebase backend hardening

---

## Findings

### CRITICAL

**C1 — Firestore rules do not enforce `createdBy == auth.uid` on create**
`firestore.rules` item/feature create rules only check that `name` is a non-empty string. A client can write any value for `createdBy`, e.g. another user's UID. This means:
- User A could frame User B as creator of an item, causing confusion about who owns what
- The delete rule (`resource.data.createdBy == request.auth.uid`) depends entirely on `createdBy` being honest — if it's spoofed, the wrong user gets delete rights
Fix: Add `request.resource.data.createdBy == request.auth.uid` to both `items` and `features` create rules.

**C2 — No uniqueness enforcement on item/feature names**
`RemoteItemRepository.createItem` and `RemoteFeatureRepository.createFeature` both call `db.collection.add(...)` without any duplicate-name check. Two users can simultaneously create "Laptop" twice with no error. The UX shows no duplicate warning.
Fix: Before `add(...)`, query `whereEqualTo("nameLower", name.trim().lowercase())`. If the collection already contains that name, throw `IllegalArgumentException` so the dialog shows an error.

---

### WARNING

**W1 — No name length limit in Firestore rules**
The rules only check `name.size() > 0`. A client could write a 10,000-character name string. Should add `name.size() <= 100` (or a reasonable limit).

**W2 — `createdAt` is not validated as a server timestamp in rules**
A client could write any value for `createdAt`, potentially corrupting the sort order. Firestore rules cannot validate `request.time == request.resource.data.createdAt` directly, but we could block the field from being user-supplied. For now, acceptable at prototype scale.

**W3 — Evaluation `orderedItemIds` not validated**
Rules allow writing an empty `orderedItemIds` list. Should add `request.resource.data.orderedItemIds.size() > 0` to the write rule.

**W4 — Uniqueness check is not atomic (race condition)**
The check-then-create pattern in the repository is NOT atomic. Two concurrent creates of "Laptop" could both pass the check before either writes to Firestore. A production fix would use a Firestore transaction with a name-index document. For this prototype, the race window is small and acceptable.

---

### MINOR

**M1 — `nameLower` field added but old documents won't have it**
Items/features created before this change won't have `nameLower`, so they won't appear in the uniqueness query. Acceptable since this is an emulator-based prototype — clearing emulator data resets to a clean state.
