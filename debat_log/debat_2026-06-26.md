# Debator Review — 2026-06-26

## Summary

The tests cover the happy path and the most obvious edge cases well, but three real gaps remain: no test verifies that the ScoreMatrix is fully populated for every item×feature pair, the `minVoteThreshold = 0` path silently produces `NaN` scores and is untested, and `ScoreAggregatorTest` reuses the same `userId = "u"` in multi-evaluation tests, which masks a duplicate-voter scenario that the aggregator does not guard against.

---

## Findings

### CRITICAL — ScoreMatrix completeness is never asserted

**File:** `DefaultFeatureScoringEngineTest.kt`

Every test asserts individual cells (`matrix.scores["A"]!!["f1"]!!`), but no test ever checks that the matrix contains an entry for **every** item × every feature. If the engine silently dropped an item or a feature from the output maps, all existing spot-checks would still pass.

Example: if the loop in `DefaultFeatureScoringEngine.kt` (line 20, `for (itemId in allItemIds)`) had a filtering bug that skipped item "B", the assertions in "three identical evaluations" and "two features are computed independently" would both pass unchanged because neither test reads `matrix.scores["B"]` in those paths.

**Suggested fix:** Add a completeness assertion to at least one test:

```kotlin
for (itemId in items) {
    assertTrue("scores missing item $itemId", matrix.scores.containsKey(itemId))
    for (featureId in features) {
        assertTrue(
            "scores[$itemId] missing feature $featureId",
            matrix.scores[itemId]!!.containsKey(featureId)
        )
        assertTrue(
            "voteCounts[$itemId] missing feature $featureId",
            matrix.voteCounts[itemId]!!.containsKey(featureId)
        )
    }
}
```

The "empty evaluations produce null scores and zero vote counts" test is the natural place for this, since it already loops over all items and features.

### CRITICAL — minVoteThreshold = 0 silently produces NaN scores; not tested

**Files:** `ScoreAggregator.kt` line 26, `DefaultFeatureScoringEngine.kt` line 18

`ScoreAggregator.aggregate` returns `scores.average()` when `scores.size >= minVotes`. When `minVotes = 0` and no one has evaluated an item, `scores` is an empty list, and `[].average()` in Kotlin returns `Double.NaN`. That NaN propagates through `ScoreRemapper.remap` and ends up stored in the `ScoreMatrix` as `NaN` — a value that is neither null (so callers won't use the null-guard path) nor a valid score.

No test exercises `minVotes = 0` at all in either `ScoreAggregatorTest` or `DefaultFeatureScoringEngineTest`.

**Why it matters:** Any UI code that formats or compares a `Double?` value will receive a non-null `NaN` and display garbage (or crash on a `!!` access expecting a valid number), while the null-guard path it was designed to use sits unused.

**Suggested fix:** Add a dedicated test:

```kotlin
@Test
fun `minVotes zero with no evaluations does not produce NaN`() {
    val result = ScoreAggregator.aggregate(emptyList(), listOf("A"), minVotes = 0)
    val score = result["A"]
    // Either null or a finite Double — never NaN
    if (score != null) assertFalse(score.isNaN())
}
```

This test will currently **fail**, which is the point — it exposes the bug so the source can add a `scores.isEmpty()` guard.

### CRITICAL — Boundary test (N == minVotes) is present but unnamed; exact-threshold semantics are ambiguous

**File:** `ScoreAggregatorTest.kt` lines 26-32 and 19-23

The boundary where `evaluations.size == minVotes` is exercised by "three identical evaluations produce correct averages" (3 evals, `minVotes = 3`), which passes because `3 < 3` is false. However:

1. The test name gives no hint that it is the threshold-boundary test. A reader looking for the boundary case will not find it.
2. There is no test for `minVotes = 3` with exactly **2** evaluations alongside a test for **3** in the same file that makes the contract explicit: the threshold is _inclusive_ (N ≥ threshold passes). If the source were changed to `<=` it would flip the two-eval test but leave the three-eval test broken in a way that is hard to diagnose.

**Suggested fix:** Rename the existing three-eval test to include "at threshold" in its name, and add one test that is explicitly one below and one at the boundary:

```kotlin
@Test
fun `exactly minVotes evaluations meets threshold and returns non-null scores`() { ... }

@Test
fun `one below minVotes returns null scores`() { ... }  // already exists as "two evaluations below threshold"
```

### MINOR — Same userId repeated across evaluations in multi-vote tests

**File:** `ScoreAggregatorTest.kt` lines 27, 49, 57; `DefaultFeatureScoringEngineTest.kt` lines 29, 40-41

The `eval()` helper in `ScoreAggregatorTest` always uses `userId = "u"`. The three-vote tests therefore simulate the same user voting three times, which is not a realistic scenario. `ScoreAggregator` does not deduplicate by userId (by design), so the tests pass — but they do not verify the more realistic case of three *distinct* users.

More importantly, if a future requirement adds deduplication (each user counts once), these tests would need to be rewritten. Making distinct user IDs explicit now (`"u1"`, `"u2"`, `"u3"`) costs nothing and makes intent clear.

The `DefaultFeatureScoringEngineTest.kt` helper already does use `"u$it"` (line 13 eval call with `userId: String`), so this inconsistency only affects `ScoreAggregatorTest`.

### MINOR — `ScoreRemapper.remapOrNull` tested only for null and negative input

**File:** `ScoreRemapperTest.kt` lines 24-27

`remapOrNull` is tested for `null → null` and `-1.0 → 0.0`. There is no test for a positive or zero value through `remapOrNull`. The delegation to `remap` is one line, so the risk is low, but the two-argument form is what `DefaultFeatureScoringEngine` actually calls — exercising it with a full range value (e.g. `+1.0 → 10.0`) would close the gap.

---

### MINOR — EvaluationConverter not tested with duplicate item IDs

**File:** `EvaluationConverterTest.kt`

`EvaluationConverter.convert` ends with `.toMap()`, which silently drops duplicate keys (last write wins). Passing `listOf("A", "A")` produces `{"A": -1.0}` — a one-entry map — instead of a two-entry map. The second "A" overwrites the first and gets the lower score. Whether this is correct behavior is undefined by the spec, but it is an unguarded path that returns a silently truncated result. At minimum, the behavior should be documented; ideally a test pins it so any future change is deliberate.

---

### SUGGESTION — Test names do not encode expected output values

**Files:** All four test files

Test names like `three identical evaluations produce correct display scores` pass the smell test but make CI failure messages ambiguous — "correct" is defined only in the test body. Names like `three identical evals at threshold produce A=10 B=5 C=0 display scores` are more verbose but let you triage a red build without opening the file. This is purely a style call.

---

### SUGGESTION — No test for a feature key present in evaluationsByFeature but absent from allFeatureIds

**File:** `DefaultFeatureScoringEngineTest.kt`

The engine iterates `allFeatureIds`, not `evaluationsByFeature.keys`. If a caller passes evaluations for `"f3"` but does not include `"f3"` in `allFeatureIds`, those evaluations are silently ignored. No test documents this behavior. Low risk, but worth a single test to lock in the contract.

## Verdict

APPROVE WITH FIXES

Fix the three CRITICALs before merge:
1. Add a completeness assertion (every item × every feature is present in both maps).
2. Add a `minVotes = 0` test — it will expose a real NaN bug in `ScoreAggregator` that must be fixed in source.
3. Rename the implicit threshold-boundary test and add a second explicit "one below" test so the inclusive/exclusive contract is unambiguous.
