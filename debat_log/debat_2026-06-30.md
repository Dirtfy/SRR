# Debat Log — 2026-06-30

## Scope
Two changes reviewed together:
1. Cascade delete: item/feature deletion did not modify server-side evaluations
2. New feature: item image update by the item creator

---

## Findings

### CRITICAL — Delete item leaves stale item IDs in all evaluations
**File:** `RemoteItemRepository.kt` / `UserViewModel.kt`

`deleteItem(id)` only removed the `items/{id}` Firestore document. Every evaluation's
`orderedItemIds` list still contained the deleted item's ID. The scoring engine would
then try to rank a non-existent item, producing corrupt or missing scores.

**Fix:** After `deleteItem` succeeds, `removeItemFromEvaluations(itemId, allFeatureIds)`
is called to strip the deleted ID from every evaluation subcollection across all features.

---

### CRITICAL — Delete feature orphans its evaluation subcollection in Firestore
**File:** `RemoteFeatureRepository.kt` / `UserViewModel.kt`

`deleteFeature(id)` only removed `features/{id}`. The subcollection at
`evaluations/{featureId}/userEvaluations/*` was never deleted — Firestore does not
auto-delete subcollections. Stale user evaluation documents accumulated indefinitely.

**Fix:** After `deleteFeature` succeeds, `deleteEvaluationsForFeature(featureId)` is
called to batch-delete all documents in the orphaned subcollection.

---

## New Feature Review

### Image update (no issues found)
- `updateItemImage` uses a targeted Firestore field update (not a full document overwrite),
  so other fields (name, createdBy, createdAt) are safe.
- The edit button is shown only when `item.createdBy == currentUserId`, matching the
  existing delete-button ownership guard.
- Upload flow reuses the same `storageRepository.uploadItemImage` path as `AddItemDialog`.
- The confirm button is disabled until upload completes (`imageUrl != null`), preventing
  a save with a local URI instead of the final storage URL.

**Verdict:** No issues. Feature is safe to ship.
