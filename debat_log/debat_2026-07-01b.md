# Debat Log — 2026-07-01b

**Debator:** Claude
**Scope:** Client-side image resizing before Firebase Storage upload

## Changes Reviewed

### `core/util/ImageResizeCalculator.kt` (new)
Pure Kotlin `planResize(origWidth, origHeight, maxDimension)` returns a `ResizePlan`
(sampleSize, targetWidth, targetHeight). No Android imports — fully unit-testable.
Edge cases guarded by `require()`.

### `remote/repository/RemoteStorageRepository.kt`
- `uploadItemImage` now calls `encodeAsJpeg(uri)` inside `withContext(Dispatchers.IO)`.
- `encodeAsJpeg`: reads EXIF rotation → bounds-only decode → delegates to `planResize` →
  sub-sample decode → applies EXIF rotation via `Matrix.postRotate` → scales to exact
  plan dimensions → compresses to JPEG at 85 % quality → returns `ByteArray`.
- Upload switched from `putFile` to `putBytes` so the (already-in-memory) resized bytes
  are sent directly without a second file I/O pass.

### `storage.rules`
10 MB server-side guard restored. Client-side resize caps output at ~1–3 MB
(JPEG 85 %, max 1920 px), so the guard is always satisfied. Keeps defense-in-depth.

## Findings

### CRITICAL
None.

### WARNING
- **W1 — No unit tests for `planResize`:** The pure-Kotlin function is fully testable
  but no tests were added in this commit. Not blocking (no coverage regression), but
  should be added soon.

### MINOR
- **M1 — EXIF rotation stream reopened separately from bounds-pass stream:** Three
  `openInputStream` calls total per upload. ContentResolver streams are not seekable,
  so this is the correct approach. Slight overhead is acceptable.
- **M2 — `inSampleSize` slightly under-approximates due to integer halving:**
  `planResize` halves `w/h` by integer division; the actual decoded size from
  `BitmapFactory` may differ by a pixel. The subsequent `createScaledBitmap` to exact
  plan dimensions corrects this.

## Decision

No CRITICALs. Proceed.
