## Topic 1: Scoring Algorithm Edge Cases

### 1a — EvaluationConverter: n=1 returns 0.0

**Problem.** For n=1, the plan special-cases `if (n == 1) return mapOf(orderedItemIds[0] to 0.0)`. The score 0.0 maps to 5.0 after remapping — indistinguishable from a genuinely neutral item when 3+ single-item evaluations accumulate.

**Recommended fix.** Reject single-item submissions at the UI layer ("Please rank at least 2 items to submit"), preserving semantic consistency without a special-case in the converter.

**Severity:** Medium

---

### 1b — ScoreAggregator: Unknown item IDs silently discarded

**Problem.** `collectedScores[itemId]?.add(score)` uses a safe call. If `orderedItemIds` contains a deleted/unknown item ID, the entry is silently dropped with no warning. Scores for known items are not corrupted, but data integrity issues are invisible.

**Recommended fix.** Log a warning when `collectedScores[itemId]` is null: `Log.w("ScoreAggregator", "Unknown itemId=$itemId in evaluation userId=${evaluation.userId}")`.

**Severity:** Low

---

### 1c — N=3 threshold hardcoded in LoadFeatureScoresUseCase

**Problem.** `minVoteThreshold = 3` is hardcoded at the call site inside `execute()`, bypassing the interface's intent to allow configurability.

**Recommended fix.** Add `val minVoteThreshold: Int = 3` as a constructor parameter on `LoadFeatureScoresUseCase`.

**Severity:** Low

## Topic 2: N+1 Firestore Reads in LoadFeatureScoresUseCase

**Problem.** `features.associate { feature -> feature.id to evaluationRepository.getEvaluationsForFeature(feature.id).getOrThrow() }` executes F sequential Firestore round-trips. On mobile (~100ms per call), 10 features = ~1 second of serial network time.

**Recommended fix (parallel coroutines):**
```kotlin
val evaluationsByFeature: Map<String, List<Evaluation>> = coroutineScope {
    features.associate { feature ->
        feature.id to async {
            evaluationRepository.getEvaluationsForFeature(feature.id).getOrThrow()
        }
    }.mapValues { (_, deferred) -> deferred.await() }
}
```

**Severity:** High

## Topic 3: Full Loading Flash on Submit

**Problem.** `onSubmitEvaluation()` on success calls `loadAllData()` which sets `_uiState.value = UserUiState.Loading`, destroying all Ready state and producing a full-screen flash after every submission.

**Recommended fix.** Add `val isRefreshing: Boolean = false` to `UserUiState.Ready`. On submit success: close the editor, set `isRefreshing = true`, reload, then set `isRefreshing = false`.

**Severity:** Medium

## Topic 4: No-DI Factory Pattern — Long-Term Maintainability

**Problem.** Each ViewModel factory constructs all concrete dependencies inline — `RemoteXRepository()` called directly inside the factory. Impossible to inject mocks for unit testing without hitting the emulator.

**Verdict.** Acceptable for prototype scope. Document the unit-testing limitation explicitly.

**Severity:** Low (now), Medium (later)

## Topic 5: Tab Switch Does Not Clear Stale Selection State

**Problem.** `onTabSelected()` only changes `activeTab`. If a user opens an item, switches to Features tab (item detail still set in state), opens a feature, then presses back twice — they return to an item detail they thought they had left.

**Recommended fix:**
```kotlin
fun onTabSelected(tab: UserUiState.Tab) {
    (_uiState.value as? UserUiState.Ready)?.let {
        _uiState.value = it.copy(
            activeTab = tab,
            selectedItem = null,
            selectedFeature = null,
            evaluationEditor = null
        )
    }
}
```

**Severity:** Medium

---

## Topic 6: EvaluationEditorState Not Nested Inside Ready

**Problem.** `EvaluationEditorState` is a peer of `Loading`, `Error`, `Ready` in the sealed class. A developer could accidentally write `_uiState.value = UserUiState.EvaluationEditorState(...)` and the compiler would accept it silently.

**Recommended fix.** Nest `EvaluationEditorState` inside `Ready`:
```kotlin
data class Ready(...) : UserUiState() {
    data class EvaluationEditorState(...)
}
```

**Severity:** Medium

---

## Topic 7: isAlreadySignedIn() Fires on Every Recomposition

**Problem.** `LaunchedEffect(Unit) { if (viewModel.isAlreadySignedIn()) onLoginSuccess() }` fires again on rotation (Fragment recreated). `onLoginSuccess()` → `navigateToMine()` a second time → fresh UserFragment with fresh ViewModel — brief re-Loading flash.

**Recommended fix.** Check `isAlreadySignedIn()` in `LoginViewModel.init {}` and store result in `LoginUiState`. Consume the same way as `isLoginSuccess`.

**Severity:** Low

---

## Topic 8: Sign-Out During Active Data Load

**Problem.** Sign-out button is enabled during `Loading` state. `Firebase.auth.signOut()` is called while Firestore reads are in flight. Coroutines are cancelled safely via `viewModelScope` cancellation on `onCleared()`, but could produce logcat error noise from in-flight requests completing with permission-denied.

**Recommended fix.** Disable sign-out button when `uiState is Loading`:
```kotlin
IconButton(
    onClick = { (activity as? MainActivity)?.signOut() },
    enabled = uiState !is CompilationUiState.Loading
) { ... }
```

**Severity:** Low

---

## Topic 9: Partial Evaluations Use n-Item Scale

**Problem.** A user who evaluates 3 out of 5 items gets scores on a 3-item scale. Middle items in a partial evaluation receive the same raw score regardless of how many items were skipped — a known limitation of uniform-spacing.

**Recommended fix.** Document this in CLAUDE.md; no code change needed.

**Severity:** Informational

---

## Topic 10: No Layer Rule Enforcement Tooling

**Problem.** No mechanism enforces that `core/` never imports Firebase/Android APIs. A stray import in `DefaultFeatureScoringEngine` would break unit testability silently.

**Recommended fixes (in order of cost):**
1. Add a "Layer rules" section to CLAUDE.md listing forbidden imports per layer.
2. Add a custom Android lint `ForbiddenImport` rule for `core/` package.
3. Split into separate Gradle modules (gold standard, significant restructure).

**Severity:** High
