# Debator Review — 2026-06-26c

## Summary

Two changes are under review: `TabRow` replaced with `PrimaryTabRow` in `user/Screen.kt` (line 130) and `compilation/Screen.kt` (line 124). The replacement itself is correct — the API is identical and the import is already covered by the wildcard `androidx.compose.material3.*`. No arguments were dropped. However, the broader audit surfaces two CRITICAL issues: (1) `onMapItemTap` is accepted and threaded through `CompilationScreen` and `CompilationReadyContent` / `CompilationTabContent` but is never forwarded into `CompilationMapTab`, so tapping a map point can never update `CompilationUiState.mapPopupItem` — the popup in `MapScreen` runs entirely on local `remember` state and is invisible to the ViewModel; and (2) the back-button logic in `CompilationFragment` checks `state.mapPopupItem` and calls `viewModel.clearSelection()`, but `mapPopupItem` is always `null` at the ViewModel level, so that branch is dead code. One MINOR issue and two suggestions follow.

---

## Findings

### CRITICAL — `onMapItemTap` is silently dropped: `mapPopupItem` is always `null`

**Trace:**

- `CompilationFragment.ScreenContent` passes `onMapItemTap = viewModel::onMapItemTap` to `CompilationScreen` — correct.
- `CompilationScreen` passes `onMapItemTap` to `CompilationReadyContent` — correct.
- `CompilationReadyContent` passes `onMapItemTap` to `CompilationTabContent` — correct.
- `CompilationTabContent` (line 134–138) routes `CompilationUiState.Tab.MAP` to:
  ```kotlin
  CompilationUiState.Tab.MAP -> CompilationMapTab(state, onMapXFeatureSelected, onMapYFeatureSelected)
  ```
  `onMapItemTap` is **not** passed here. `CompilationMapTab`'s signature (lines 258–261) does not accept it.
- `MapScreen` manages its popup entirely through `var selectedPoint by remember { mutableStateOf<Item?>(null) }` (line 28 of `map/Screen.kt`). Tapping a point sets this local state only.

Consequence: `CompilationViewModel.onMapItemTap()` is never called at runtime. `CompilationUiState.Ready.mapPopupItem` is always `null`. The popup on the map works visually (local state), but:
1. The ViewModel's `clearSelection()` has a dead branch — `state.mapPopupItem != null` is always false.
2. `CompilationFragment.shouldShowBackButton()` checks `state.mapPopupItem != null` (line 38) — this is always false, so the back button never appears while the map popup is open.
3. `CompilationFragment.onBackClick()` checks `state.mapPopupItem != null` (line 43) — same dead branch.
4. Configuration change / process death while a map popup is open will silently dismiss the popup because it lives only in local `remember` state, not in `StateFlow`.

Fix: Add `onMapItemTap: (Item) -> Unit` parameter to `CompilationMapTab`, and inside `MapScreen` wire `onPointClick` to call the outer callback in addition to (or instead of) updating `selectedPoint`. Alternatively, lift the popup entirely into `CompilationReadyContent` driven by `state.mapPopupItem`, but that requires `MapScreen` to suppress its local popup when an external selection exists.

### CRITICAL — Back-button "mapPopupItem" branches in `CompilationFragment` are dead code

This is a direct consequence of the first finding, but is noted separately because it is a correctness issue in the Fragment layer that would survive even a partial fix.

`CompilationFragment.shouldShowBackButton()` (line 38) and `onBackClick()` (line 43) both test `state.mapPopupItem != null`. Since `mapPopupItem` is never written by the ViewModel (see above), these branches never execute. The back-button does not appear when the map popup is open. If a developer later fixes the `onMapItemTap` wiring but forgets that the popup dismissal also needs to be wired through `clearSelection()`, the Fragment-level back-button will still fail to dismiss the popup.

### MINOR — `onMapItemTap` propagation compiles with unused parameter warning risk

`CompilationTabContent` accepts `onMapItemTap: (Item) -> Unit` as a parameter (line 119) but does not use it (the call at line 137 omits it). In Kotlin this is a warning-level issue, not a compile error, but if `@Suppress("UNUSED_PARAMETER")` is not present and the project enables `-Werror`, the build will fail. Even without `-Werror`, the IDE will flag the unused parameter, which creates noise and is misleading — `onMapItemTap` looks intentional but has no effect.

### SUGGESTION — `PrimaryTabRow` replacement is correct but context should be documented

`PrimaryTabRow` is the Material 3 successor to `TabRow` and has an identical `selectedTabIndex + tabs` content-lambda API. No `indicator` or `divider` lambdas were passed in either call site in the original code, so nothing was dropped. The wildcard `import androidx.compose.material3.*` already covers `PrimaryTabRow`. The replacement is safe and correct. A brief inline comment noting the deprecation reason (`TabRow` is deprecated in M3 in favour of `PrimaryTabRow` / `SecondaryTabRow`) would help reviewers in future diffs.

### SUGGESTION — `onRetryTap` wiring is correct for both screens

`UserFragment.ScreenContent` (line 72) passes `onRetryTap = viewModel::onRetryTap`. `UserViewModel.onRetryTap()` (line 55) delegates to `loadAllData()`, which transitions to `Loading` and re-fetches. The `UserUiState.Error` branch in `UserScreen` (lines 42–52) renders a visible "Retry" button that calls `onRetryTap`. The wiring is complete and correct — no dead end. Same pattern applies to `CompilationFragment` / `CompilationViewModel`. Both are fully wired.

`EvaluationEditorSheet` (lines 282–366 of `user/Screen.kt`): the Submit button is correctly disabled while `editor.isSaving == true` (line 355, `enabled = !editor.isSaving`) and shows a `CircularProgressIndicator` in place of the label. The save error is displayed at lines 345–352 via `editor.saveError?.let { err -> Text(...) }`. Both behaviours are present and correct.

## Verdict

**APPROVE WITH FIXES** (for the `PrimaryTabRow` change itself)

The `TabRow` → `PrimaryTabRow` replacement is correct, complete, and safe to commit. The API surface is identical; no parameters were dropped; the import is already in scope.

However, the broader audit reveals that the map-popup wiring has been broken since the `MapScreen` refactor and has not been caught because visual testing of the popup works (local `remember` state renders correctly). Two CRITICAL issues must be fixed before Phase 4 is considered done:

Required before next commit:
1. **Fix `onMapItemTap` wiring** — pass `onMapItemTap` through `CompilationMapTab` into `MapScreen`, replacing or supplementing the local `selectedPoint` state so that `viewModel.onMapItemTap()` is called when a point is tapped.
2. **Verify `clearSelection()` dismisses the map popup** — once `mapPopupItem` is live in ViewModel state, confirm that `CompilationFragment.onBackClick()` correctly calls `viewModel.clearSelection()` and that `shouldShowBackButton()` returns `true` while the popup is open. The Fragment logic is already written correctly for this (lines 38 and 43); it just has no effect until the ViewModel state is actually set.
