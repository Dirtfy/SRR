## Root causes found

### CRITICAL 1 — No UI feedback after successful evaluation submit

After `submitEvaluation` succeeds, `loadAllData()` runs and the UI returns to the feature detail view. Because `minVoteThreshold = 3`, all scores remain `—` until 3 users have evaluated. There is no message, toast, badge, or any other indicator that the evaluation was saved. To the user it looks identical to before — "nothing happened."

**Fix:** Add `evaluatedFeatureIds: Set<String>` to `UserUiState.Ready`, populated from `cachedEvaluations` in `loadAllData()`. In `UserFeatureDetailContent`, when `hasEvaluated == true` show a banner: "Your evaluation is saved. Scores appear once 3 users have evaluated."

### CRITICAL 2 — Stale `editor` reference on submit failure (missed W1)

`onSubmitEvaluation` restores stale captured `editor` on failure (same pattern as W1 fixed in `onAddItem`/`onAddFeature`):
```kotlin
// WRONG — uses old captured `editor`, loses any name typed during save
_uiState.value = s.copy(evaluationEditor = editor.copy(isSaving = false, saveError = e.message))
```
Should read the current editor from `_uiState.value` instead.

### WARNING — Debug build always routes to local Firebase emulator

`SRRApplication.kt` unconditionally calls `useEmulator("localhost", ...)` in all debug builds. This means manual device testing requires ADB reverse ports to be active at all times. When `installDebug` runs, ADB resets the reverse ports — causing the exact "submit not working" symptom reported (reads served from Firestore offline cache; writes queued but never sync). Fix: document this more prominently, or re-apply reverse ports automatically as part of the install script.
