### WARNING

**W1 — Firestore emulator HTTP DELETE is unreliable for within-session isolation**
`clearEmulator()` sends `DELETE` to `localhost:8080/emulator/v1/...` via ADB reverse. The endpoint returns 200 before deletions are committed internally. This means `@Before` clear does NOT guarantee a clean slate before each test.

Mitigation in place: timestamp-prefix emails (Auth isolation) + unique featureId prefixes (evaluation path isolation) + presence assertions instead of count assertions for global collections (items/features). This is sufficient for the current test suite. A more robust future improvement would be to use unique Firestore collection prefixes per test, but that would require changing repository interfaces.

**W2 — `firestoreSettings.setPersistenceEnabled(false)` silently fails after first call**
If the Firestore instance is already initialized by a prior test class, setting `firestoreSettings` again throws and is swallowed by the try-catch. The first class to run sets persistence off; subsequent classes' `@BeforeClass` silently fail but the setting is already applied from the first call. This is brittle — if test class execution order changes, the setting might not apply before the first Firestore operation.

Acceptable for now given the JVM process initialization order is consistent (alphabetical per class name) and the first class alphabetically (`CompilationScreenUiTest`) does not use Firebase.

---

### MINOR

**M1 — `deleteUrl` helper duplicated across 3 test classes**
Each of `RemoteEvaluationRepositoryTest`, `RemoteItemFeatureRepositoryTest`, and `MultiUserFlowTest` defines its own identical `deleteUrl(urlStr)` private function. Could be a shared test utility object. Not worth refactoring now — test files are self-contained and isolated.

**M2 — `clearEmulator()` called in both `@Before` and `@After`**
The `@After` clear is redundant given `@Before` already clears before the next test. Kept for safety (leaves emulator clean after the test class finishes). No correctness concern.
