# Debat Log — 2026-06-27d

**Role:** Debator  
**Scope:** Debug auto-login feature

## Design

- `LoginViewModel.debugAutoLogin()`: guarded by `BuildConfig.DEBUG`. Tries `signIn(DEBUG_EMAIL, DEBUG_PASSWORD)`; on any failure tries `signUp(...)` (covers fresh emulator where account doesn't exist yet). Constants hardcoded as `private const val` in the companion object.
- `LoginScreen.LaunchedEffect(Unit)`: if not already signed in, call `debugAutoLogin()` instead of doing nothing.
- Auto-launch: `adb shell am start` after `installDebug`.

## Findings

### CRITICAL
None.

### WARNING

**W1 — `signUp` fallback swallows wrong-reason failures**  
If `signIn` fails for a network/transient reason (not user-not-found), the fallback `signUp` call will hit `FirebaseAuthUserCollisionException` (account already exists) and show a confusing error. In the local emulator this is very unlikely but possible.

**Decision:** Accept. In the emulator environment a failed `signIn` almost always means the account doesn't exist yet (emulator was just cleared). The error message would still appear in the UI, making it diagnosable.

### MINOR

**M1 — Debug credentials in source**  
Hardcoded `debug@srr.dev` / `debug1234` are visible in version control. Acceptable for emulator-only debug builds — these credentials are meaningless against production Firebase.
