# Debator Review — 2026-06-26b

## Summary

Three changes are under review: (1) the Firestore path in `RemoteEvaluationRepository.kt` was changed from `features/` to `evaluations/`; (2) the Firestore security rules were updated to match the new path; (3) a set of sub-screen `Screen.kt` files were deleted as dead code. The path fix and the rules update are both correct and consistent with the README. The dead-code deletion claim is mostly accurate, but leaves orphaned `Item.kt` files behind and exposes a separate pre-existing bug in the map screen that the change does not fix.

---

## Findings

### CRITICAL — `MapScreen` axis state is local, not wired to `CompilationUiState`

`compilation/map/Screen.kt` manages `featureX` and `featureY` as **local `remember` state** (lines 27–28). `CompilationMapTab` in `compilation/Screen.kt` (lines 258–287) calls `MapScreen(items = mapItems, availableFeatures = featureNames)` and passes `onMapXFeatureSelected` / `onMapYFeatureSelected` to itself, but those two callbacks are never forwarded into `MapScreen`. `MapScreen` does not accept them as parameters at all.

Consequences:
- `CompilationUiState.Ready.mapXFeatureId` and `mapYFeatureId` are always `null`; axis selection is silently ignored by the ViewModel.
- On configuration change or process death, the selected axes reset to defaults because they are only in `remember` state, not in `StateFlow`.
- The axis-selection dropdowns inside `MapScreen` work visually but are disconnected from the application state machine described in the README (Step 19).

This is a pre-existing architectural inconsistency. The dead-code deletion change does not introduce it, but the review brief mentions that `compilation/map/Screen.kt` was **kept** specifically because `compilation/Screen.kt` imports it — so if the map files are considered production-ready, this disconnect must be noted as a blocker for Phase 4 correctness.

### CRITICAL — `EvaluationRecord` shape diverges from `RemoteEvaluationRepository` usage

`remote/model/EvaluationRecord.kt` declares four fields: `userId`, `featureId`, `orderedItemIds`, `submittedAt`. However `RemoteEvaluationRepository.submitEvaluation()` constructs the record with only three of those fields: `userId`, `featureId`, and `orderedItemIds` — it does **not** write `submittedAt`. The README's canonical Firestore document shape (Step 2) includes `submittedAt: Timestamp`, and the README's Step 14 snippet passes `submittedAt = Timestamp.now()` in the `hashMapOf`. The current implementation omits it.

Additionally, `getEvaluationsForFeature()` reads `userId` via `doc.getString("userId")` (line 40), but the README's reference implementation (Step 14) reads it as `doc.id` (the document ID is the userId). Reading from the field and reading from the document ID are both valid approaches if the field is always written, but the write path in `submitEvaluation()` does write `userId` as a field (line 23) — so this is consistent within itself. The inconsistency is the missing `submittedAt` on writes.

Risk: `submittedAt` is absent from every document written by the app, breaking any future query that sorts or filters by submission time.

### MINOR — Orphaned `Item.kt` UI model files remain after Screen deletion

The following files were claimed as deleted (dead code), but only the `Screen.kt` files were removed. All corresponding `Item.kt` UI model files still exist on disk and compile into the APK:

- `user/items/Item.kt` — defines `data class Item(val title: String, val imageRes: Int)`
- `user/items/detail/Item.kt` — not verified directly but directory exists
- `user/features/Item.kt` — defines `data class Item(val id: Int, val name: String, val ratedCount: Int, val totalCount: Int)`
- `user/features/detail/Item.kt` — directory exists
- `compilation/items/Item.kt` — exists
- `compilation/items/detail/Item.kt` — exists
- `compilation/features/Item.kt` — defines `data class Item(val id: Int, val name: String, val totalCount: Int)`
- `compilation/features/detail/Item.kt` — exists

None of these types are imported anywhere (confirmed by grep across the entire `ui/` source tree). They are dead compiled classes that inflate APK size marginally and create confusion: the comment in `UserUiState.kt` (line 12–13) explicitly warns "NOT the UI models in user/items/Item.kt or user/features/Item.kt", which means these files are actively misleading to future developers. They should have been deleted alongside the Screen files.

### MINOR — `MapItemPopup` uses hardcoded feature labels "Safety" / "Comfort"

`compilation/map/Popup.kt` lines 66–67 call `ScoreBadge(label = "Safety", ...)` and `ScoreBadge(label = "Comfort", ...)`. These are hardcoded strings that do not reflect the actual selected X and Y feature names from `CompilationUiState`. Since `MapScreen` already receives an `availableFeatures: List<String>` parameter but does not pass the selected feature names through to `MapItemPopup`, the popup will always show "Safety" and "Comfort" regardless of what features the user selected. This is a cosmetic bug but causes incorrect labeling in production.

### MINOR — `firestore.rules` does not restrict reads on the `evaluations/{featureId}` document itself

The rule covers the subcollection path `match /evaluations/{featureId}/userEvaluations/{userId}` but there is no rule for `match /evaluations/{featureId}`. In Firestore security rules, a match on a subcollection document does **not** automatically grant access to the parent document. However, a client that calls `db.collection("evaluations").get()` to list the top-level collection could enumerate all `featureId` values without being blocked, because no `allow read` or `allow list` denial is stated for `/evaluations/{featureId}`. In practice, Firestore denies reads not covered by any rule, so listing `evaluations` is implicitly denied. This is technically safe but worth documenting explicitly for clarity, especially since the README's Step 6 template rule also does not add an explicit denial here. No code change is strictly required, but a comment in the rules file noting the intentional implicit denial would prevent future confusion.

### SUGGESTION — `CompilationMapTab` converts scores to `[-1, +1]` needlessly

`compilation/Screen.kt` lines 276–277 convert `[0, 10]` display scores back to `[-1, +1]` raw range before passing them to `MapItem`. Then `compilation/map/Screen.kt` lines 171–172 converts them back from `[-1, +1]` to `[0, 1]` canvas fraction using `(xVal + 1f) / 2f`. This round-trip is correct but indirect: dividing the `[0, 10]` display score by 10 directly would produce the same `[0, 1]` canvas fraction with no intermediate conversion. The current approach is not wrong, but it is fragile — if either conversion formula changes independently, the plot positions will silently drift.

### SUGGESTION — `EvaluationEditorState` is correctly nested in `Ready` (acknowledged positive)

`UserUiState.EvaluationEditorState` is declared as a nested class inside `Ready` (lines 22–28 of `UserUiState.kt`), not as a sibling sealed subclass. This is correct and aligns with the comment at line 22 ("nested inside Ready so it cannot be accidentally assigned to StateFlow<UserUiState>"). The README's Step 7 snippet shows it as a top-level class inside `UserUiState`, which would be looser. The implementation is actually stricter and better than the README template — this is a good deviation.

## Verdict

**APPROVE WITH FIXES**

The Firestore path change (change 1) and the security rules update (change 2) are correct, consistent, and match the README schema exactly. No missed `"features"` references remain in `RemoteEvaluationRepository`. `RemoteFeatureRepository` correctly continues to read from `collection("features")`. These two changes are safe to commit.

The dead-code deletion (change 3) is incomplete: the orphaned `Item.kt` files should be deleted in the same commit to avoid leaving misleading dead types in the codebase.

Two CRITICAL findings exist but are pre-existing bugs, not introduced by this change set: the `MapScreen` axis-state disconnect and the missing `submittedAt` on writes. These must be tracked and fixed before the app is connected to real Firestore data, but they do not block committing these three specific changes.

Required before commit:
1. Delete all orphaned `Item.kt` files in the sub-screen packages (MINOR finding — the Screen deletion is incomplete without them).

Required before Phase 4 is considered done:
2. Fix `MapScreen` to accept and propagate axis callbacks into `CompilationUiState` rather than using local `remember` state.
3. Add `submittedAt = Timestamp.now()` to the `EvaluationRecord` constructor call in `submitEvaluation()`.
4. Fix `MapItemPopup` `ScoreBadge` labels to use the actual selected feature names.
