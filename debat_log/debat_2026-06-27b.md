### WARNING

**W1 — Stale dialog `name` restored on save failure** (`UserViewModel.kt:155–159`, `UserViewModel.kt:165–171`)

In `onAddItem()` and `onAddFeature()`, the `dialog` variable is captured before the coroutine launches. On failure the code does:
```kotlin
_uiState.value = s.copy(addItemDialog = dialog.copy(isSaving = false, error = e.message))
```
This restores the old `dialog` (including its `name` at the moment save was triggered). The text field is not disabled during saving (only the confirm button is), so a user who types into the field during the save will have their input overwritten with the pre-save name when an error occurs. Confusing UX but not data-corrupting.

**Decision:** Fix. The failure handler should pull the latest `dialog` state from `_uiState.value`, not the stale captured copy.

**W2 — No max-length guard on item/feature names**  
Firestore security rule validates `name.size() > 0` but not an upper bound. An extremely long name could render poorly and potentially hit Firestore field-size limits. Low probability but worth noting.

**Decision:** Defer — add a client-side `name.length <= 100` guard in a future pass.

### MINOR

**M1 — `clickableItem` extension is a no-op wrapper** (`Screen.kt:500–501`)  
The extension `private fun Modifier.clickableItem(onClick: () -> Unit) = this.clickable(onClick = onClick)` adds no behavior over plain `.clickable {}`. Readability win is marginal.

**M2 — No empty-state UI** in `ItemsTabContent` and `FeaturesTabContent`  
When no items or features exist (fresh Firestore, new project), the list shows nothing — no "Add your first item" hint. Degrades new-user onboarding but is not a bug.

**M3 — `ratedCount` computed inline per recompose** (`Screen.kt:252–255`)  
`items.count { matrix.scores[it.id]?.get(feature.id) != null }` runs on every recompose for each feature row. Negligible for small lists but worth noting.
